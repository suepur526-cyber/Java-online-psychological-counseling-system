# 基于 Java 的在线心理咨询系统需求与技术栈整理

## 一、系统概述

本系统是一个面向校园或普通心理咨询场景的在线心理咨询平台，主要用于解决传统线下心理咨询存在的预约不便、时间地点受限、信息管理不统一等问题。

系统采用 B/S 架构，用户通过浏览器访问平台。系统主要围绕普通用户、心理老师和管理员三类角色展开，实现注册登录、公告浏览、在线心理测试、心理服务预约、在线咨询、咨询回复、用户管理、公告管理和测试管理等功能。

考虑到本系统属于学生毕业设计项目，业务逻辑应以简单、完整、可实现为原则，不设计过于复杂的智能推荐、复杂实时调度和大型数据分析功能，但初稿中提到的核心需求应全部保留。

## 二、系统角色

### 1. 普通用户

普通用户是系统的主要使用者，主要完成心理健康信息查看、心理测试、预约心理老师、提交咨询内容和查看咨询回复等操作。

### 2. 心理老师

心理老师负责处理用户的预约申请、查看用户咨询内容、进行咨询回复，并维护心理测试题库等内容。

### 3. 管理员

管理员负责系统后台管理，包括用户管理、心理老师管理、公告管理、心理测试管理、预约管理、咨询管理和系统基础信息维护等。

## 三、功能需求

## 3.1 普通用户功能需求

### 3.1.1 注册登录

普通用户可以填写账号、密码、姓名、性别、邮箱、手机号等基础信息完成注册。注册成功后，用户可以通过账号和密码登录系统。

系统需要对用户登录状态进行判断，只有登录后的用户才能进行心理测试、预约和咨询等操作。

### 3.1.2 查看公告

用户可以在系统首页或公告模块中查看平台发布的公告信息，包括心理健康知识、心理疏导活动、系统通知和相关资讯等。

公告内容主要包括公告标题、简介、图片和详细内容。

### 3.1.3 在线心理测试

用户可以进入在线心理测试模块，选择系统中已经发布的心理测试题目进行作答。

用户提交答案后，系统根据题目分值自动计算测试得分，并根据分数显示相应的心理状态或测试结果。同时，系统需要保存用户的测试记录，方便用户后续查看，也方便心理老师辅助了解用户情况。

### 3.1.4 心理服务查看

用户可以查看心理老师或心理服务信息，包括老师姓名、性别、照片、费用、工作时间、执教时长、个人简介、荣誉信息等。

该功能用于帮助用户选择合适的心理老师。

### 3.1.5 在线预约

用户可以选择心理老师，并填写预约时间和预约备注，提交一对一心理咨询预约申请。

预约提交后，预约状态默认为“待审核”。心理老师审核后，预约状态可以变为“已通过”或“已驳回”，用户可以查看预约审核结果和审核回复。

### 3.1.6 在线咨询

用户可以向心理老师提交咨询内容，系统保存咨询记录。

考虑到学生项目实现难度，在线咨询可以简化为“用户提交咨询内容，老师进行回复”的留言式咨询方式。如果项目需要体现实时咨询，也可以在论文中说明使用 WebSocket，但实际实现可以以消息记录和回复为主。

### 3.1.7 咨询回复查看

用户可以查看心理老师对咨询内容的回复，包括原咨询内容、回复内容和回复日期。

### 3.1.8 留言提交

用户在咨询后可以继续提交留言，填写留言标题和留言内容。系统保存留言状态，老师回复后，用户可以查看回复结果。

## 3.2 心理老师功能需求

### 3.2.1 注册登录

心理老师可以填写账号、密码、姓名、联系方式、擅长领域、个人简介或资质信息等内容完成注册。

心理老师登录后进入老师端后台。若系统设置资质审核，则老师需要管理员审核通过后才能使用预约处理和咨询回复等功能。

### 3.2.2 服务预约管理

心理老师可以查看用户提交给自己的预约申请，包括预约编号、用户姓名、预约时间、预约备注和预约状态等。

老师可以对预约进行审核处理，审核结果包括通过或驳回，并可以填写审核回复。

### 3.2.3 在线咨询管理

心理老师可以查看用户提交的咨询内容和历史咨询记录，并根据用户情况进行处理。

系统应保存咨询内容、咨询日期、用户信息和老师信息，方便后续查询和管理。

### 3.2.4 咨询回复管理

心理老师可以针对用户提交的咨询内容填写回复内容。回复提交后，系统保存回复记录，并在用户端展示。

该功能是系统心理咨询业务闭环的重要部分。

### 3.2.5 心理测试管理

心理老师可以维护心理测试题库，包括新增、修改、删除测试题目。

测试题目可包含测试名称、试题名称、选项、分值、答案、答案解析、题目类型和排序等信息。

### 3.2.6 测试记录查看

心理老师可以查看用户心理测试记录和测试结果，用于辅助了解用户心理状态，并为咨询回复提供参考。

## 3.3 管理员功能需求

### 3.3.1 管理员登录

管理员通过后台账号和密码登录系统，登录成功后进入后台管理界面。

管理员拥有系统最高管理权限，可以管理用户、老师、公告、测试、预约和咨询等信息。

### 3.3.2 用户信息管理

管理员可以查询、修改和删除普通用户信息，也可以对异常账号进行禁用或恢复。

用户信息主要包括账号、姓名、性别、邮箱、手机号、头像、注册时间和账号状态等。

### 3.3.3 心理老师管理

管理员可以管理心理老师信息，包括查询、修改、删除心理老师资料。

如果系统包含资质审核功能，管理员可以对心理老师提交的资质或账号进行审核。审核通过后，老师可以正常提供咨询服务；审核不通过时，管理员可以填写驳回原因。

### 3.3.4 公告管理

管理员可以新增、修改、删除和查询公告信息。

公告发布后，普通用户可以在前台查看公告内容。

### 3.3.5 心理测试管理

管理员可以管理心理测试、试题和测试记录。

管理员可以查看测试题库、维护测试状态，也可以查看用户测试结果和测试记录。

### 3.3.6 服务预约管理

管理员可以查看系统中的预约信息，包括用户、心理老师、预约时间、预约状态和审核回复等内容。

该功能主要用于后台监管和数据维护。

### 3.3.7 在线咨询管理

管理员可以查看平台中的在线咨询记录和咨询回复记录，用于平台管理和异常问题处理。

考虑到隐私保护，管理员只做必要的数据监管，不参与具体咨询内容处理。

### 3.3.8 系统管理

管理员可以维护系统基础信息，例如系统参数、轮播图、关于我们、平台简介等内容。

该功能用于保证系统页面内容完整，方便后台统一维护。

## 四、非功能需求

### 4.1 性能需求

系统应支持多个用户同时进行登录、测试、预约和咨询等操作。

作为学生项目，性能需求可以设置为普通页面查询和提交操作响应较快，页面响应时间一般控制在 3 秒以内，满足日常校园心理咨询系统的使用需求。

### 4.2 安全性需求

系统需要具备基本的登录验证和角色权限控制。

普通用户、心理老师和管理员只能访问各自权限范围内的功能模块。用户密码不建议明文存储，可以采用加密或哈希方式保存。

系统还应避免常见的非法输入问题，例如 SQL 注入和恶意脚本输入。

### 4.3 易用性需求

系统界面应简洁清晰，操作流程简单。

普通用户、心理老师和管理员分别拥有不同的菜单和操作界面，用户不需要复杂培训即可完成主要操作。

### 4.4 可维护性需求

系统采用分层开发结构，将控制层、业务层、数据访问层和前端页面分开，便于后期修改、维护和扩展。

代码结构应清晰，接口命名应规范，数据库表设计应保持简单明确。

### 4.5 兼容性需求

系统采用 B/S 架构，用户通过浏览器访问。

系统应支持 Chrome、Edge、Firefox 等主流浏览器，能够在电脑端正常使用。

## 五、数据需求

### 5.1 管理员表

用于存储管理员账号、密码、角色、创建时间等信息。

### 5.2 用户表

用于存储普通用户信息，包括用户 ID、学号或账号、密码、姓名、性别、邮箱、手机号、头像和创建时间等。

### 5.3 心理老师表

用于存储心理老师信息，包括老师 ID、账号、密码、姓名、性别、联系方式、头像、个人简介、资质信息、审核状态和创建时间等。

### 5.4 心理服务表

用于展示心理老师服务信息，包括工号、老师姓名、性别、相片、费用、工作时间、执教时长、个人简介和荣誉信息等。

### 5.5 服务预约表

用于保存用户预约信息，包括预约编号、老师工号、老师姓名、用户学号、用户姓名、预约时间、预约备注、审核状态、审核回复和是否支付等。

### 5.6 在线咨询表

用于保存用户咨询内容，包括老师工号、老师姓名、用户学号、用户姓名、咨询内容和咨询日期等。

### 5.7 咨询回复表

用于保存老师回复内容，包括老师工号、老师姓名、用户学号、用户姓名、咨询内容、回复内容和回复日期等。

### 5.8 心理测试表

用于保存心理测试基本信息，包括测试 ID、测试名称、考试时长、测试状态和创建时间等。

### 5.9 试题表

用于保存心理测试题目，包括所属测试 ID、测试名称、试题名称、选项、分值、答案、答案解析、题目类型和排序等。

### 5.10 考试记录表

用于保存用户答题记录，包括用户 ID、用户名、测试 ID、测试名称、试题 ID、试题名称、用户答案、正确答案、题目得分和用户得分等。

### 5.11 公告信息表

用于保存公告信息，包括公告标题、简介、图片、详细内容和创建时间等。

### 5.12 关于我们表

用于保存平台介绍信息，包括标题、副标题、内容和图片等。

### 5.13 系统配置表

用于保存系统基础配置参数，包括参数名称、参数值和说明等。

## 六、建议采用的技术栈

### 6.1 后端技术

后端开发语言采用 Java。

后端框架建议统一使用 Spring Boot。Spring Boot 内置 Tomcat，配置简单，适合学生项目开发，也便于系统部署和运行。

数据访问层采用 MyBatis 或 MyBatis-Plus，用于完成数据库增删改查操作。

### 6.2 前端技术

前端框架采用 Vue.js。

前端通过 Axios 向后端发送请求，前后端之间使用 JSON 格式进行数据交互。

页面可以使用 Element UI 或 Element Plus 作为组件库，降低页面开发难度。

### 6.3 数据库技术

数据库采用 MySQL，用于存储用户信息、老师信息、公告信息、心理测试、试题、预约、咨询和回复等数据。

数据库管理工具可以使用 Navicat。

### 6.4 系统架构

系统采用 B/S 架构，用户通过浏览器访问系统，服务器端负责业务逻辑处理和数据库访问。

系统采用前后端分离或简单前后端结合的方式均可。若项目已有 Vue 前端，建议采用前后端分离结构。

### 6.5 登录与权限

系统采用基于角色的权限控制方式，将用户划分为普通用户、心理老师和管理员三类角色。

登录认证可以使用 Session 或 Token。如果论文中需要体现技术完整性，可以写 JWT；如果项目实现以简单为主，也可以使用 Session。

### 6.6 在线咨询实现建议

初稿中提到 WebSocket 和实时图文咨询，但对于学生项目来说，实现难度较高。

建议将在线咨询简化为留言式咨询：用户提交咨询内容，心理老师查看后进行回复，系统保存咨询记录和回复记录。

如果必须保留 WebSocket 表述，可以在论文中写成“系统预留实时咨询扩展能力”，实际功能以咨询记录和咨询回复为主。

### 6.7 开发与测试工具

开发工具使用 IntelliJ IDEA。

接口测试可以使用 Postman。

系统测试采用黑盒测试方法，重点测试普通用户、心理老师和管理员三类角色的核心功能。

## 七、简化后的核心业务逻辑

系统整体业务流程可以简化为以下步骤：

1. 普通用户注册并登录系统。
2. 用户查看公告，了解心理健康信息。
3. 用户选择心理测试并完成答题，系统自动计算分数并保存测试记录。
4. 用户查看心理老师信息，选择老师并提交预约申请。
5. 心理老师登录系统，查看预约申请并进行审核。
6. 预约通过后，用户提交咨询内容。
7. 心理老师查看咨询内容并填写回复。
8. 用户查看老师回复，完成一次咨询流程。
9. 管理员在后台维护用户、老师、公告、测试、预约和咨询等数据。

该业务流程保留了初稿中的主要功能，同时避免复杂的实时调度、智能分析和大型权限系统，更适合学生毕业设计实现。

## 八、推荐保留但简化的内容

### 8.1 心理档案

初稿中提到心理健康档案。学生项目中可以不单独做复杂档案模块，而是通过用户信息、测试记录、咨询记录和咨询回复组合形成用户心理档案。

### 8.2 数据统计分析

初稿中提到测试统计和数据分析。学生项目中可以简化为管理员查看测试记录列表，或者按测试名称、用户姓名进行查询，不必实现复杂图表分析。

### 8.3 资质审核

初稿中提到心理老师资质审核。学生项目中可以简化为管理员修改老师审核状态，包括待审核、审核通过和审核驳回。

### 8.4 支付功能

预约表中存在是否支付字段。学生项目中可以保留字段，但业务上不实现真实支付接口，只显示“未支付”或“已支付”状态即可。

## 九、总结

本系统的需求可以概括为：围绕普通用户、心理老师和管理员三类角色，构建一个具备公告浏览、心理测试、心理服务预约、在线咨询、咨询回复和后台管理功能的在线心理咨询系统。

技术栈建议统一为 Java、Spring Boot、MyBatis、MySQL、Vue.js、Axios 和 B/S 架构。系统实现时应控制业务复杂度，以功能完整、流程清晰、易于实现和便于答辩为主要目标。
